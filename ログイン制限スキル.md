このアプリに、Microsoftログイン済みユーザーをメールアドレスで制限する機能を追加・修正してください。

方針は、既存の Microsoftログイン機能を維持したまま、アプリ本体を表示する直前に許可メールアドレスリストを確認することです。

# 重要方針

* Microsoftログイン自体を止めようとしないでください。
* ログイン後、Microsoft から返ったユーザー情報を使って、アプリ本体を表示するか拒否画面を出すかを判定してください。
* 許可されていないユーザーには、白画面ではなく分かりやすい拒否画面を表示してください。
* 環境変数が未設定の場合は、全員許可にせず、設定不足としてアプリ本体を表示しないでください。
* 既存の Microsoftログイン機能、iframe内ポータル認証、業務ロジック、API処理はできるだけ変更しないでください。

# 環境変数

対象アプリのフレームワークを確認して、正しい public 環境変数名を使ってください。

Vite / React の場合:

```env
VITE_ALLOWED_AUTH_EMAILS=
```

Next.js の場合:

```env
NEXT_PUBLIC_ALLOWED_AUTH_EMAILS=
```

複数メールアドレスはカンマ区切りにしてください。

```env
VITE_ALLOWED_AUTH_EMAILS=user1@example.com,user2@example.com
```

全員許可が本当に必要な場合だけ、明示的に `*` を許可してください。

```env
VITE_ALLOWED_AUTH_EMAILS=*
```

ただし通常は `*` を使わず、個別メールアドレスを指定してください。

# Viteで必ず注意すること

Vite では、ブラウザ側で読む環境変数は `VITE_` で始まる必要があります。

`NEXT_PUBLIC_ALLOWED_AUTH_EMAILS` は Next.js 用なので、Vite アプリでは読まれません。

また、Vite の環境変数は開発サーバー起動時、またはビルド時に埋め込まれます。

そのため `.env` を変更したら、必ず以下を案内してください。

ローカル:

```powershell
Ctrl + C
npm run dev
```

本番 Netlify:

* Netlify の環境変数を変更する
* 再デプロイする

# 判定対象メールアドレス

MSAL の `account.username` だけに依存しないでください。

Microsoft Entra の設定やアカウント状態によって、以下のように複数のメール・UPN が返ることがあります。

例:

```text
user@company.onmicrosoft.com
user@company.com
```

直接アクセス時の MSAL アカウントでは、少なくとも以下を候補として確認してください。

* `account.username`
* `account.idTokenClaims.preferred_username`
* `account.idTokenClaims.email`
* `account.idTokenClaims.upn`
* `account.idTokenClaims.login_hint`

候補を小文字化・trim し、重複を除去してから、許可リストと照合してください。

ポータル iframe から `AUTH_HINT` を受け取る実装では、`loginHint` を同じ許可リストで照合してください。

# 実装の考え方

認証状態は上位の AuthWrapper / AuthProvider に集約してください。

アプリ本体の表示直前に、必ず許可チェックを通してください。

推奨する判定の流れ:

```ts
const candidates = getAccountEmailCandidates(account);
const allowedEmail = candidates.find(isEmailAllowed);

if (!allowedEmail) {
  return {
    status: "error",
    message: getForbiddenMessage(candidates[0] || "", candidates),
  };
}

return {
  status: "authenticated",
  email: allowedEmail,
  account,
};
```

React の開発中や Hot Reload 後に認証済み状態が残ることがあります。

そのため、`authState.status === "authenticated"` の分岐でも、アプリ本体を返す直前に `isEmailAllowed(authState.email)` を再確認してください。

# 拒否画面

許可されていない場合は、以下を画面に出してください。

* アプリを表示できないこと
* ログインユーザーが許可されていないこと
* 判定対象になったメール候補
* 現在の許可設定

例:

```text
アプリを表示できません

user@company.onmicrosoft.com はこのアプリの利用が許可されていません。
判定対象: user@company.onmicrosoft.com, user@company.com。
許可設定: no-one@example.invalid。
```

これにより、環境変数が反映されていないのか、Microsoft 側から返るメールが想定と違うのかを人間が判断しやすくなります。

# 設定不足時

許可メール環境変数が空の場合は、アプリ本体を表示しないでください。

例:

```text
ログイン制限設定が不足しています。
VITE_ALLOWED_AUTH_EMAILS に許可メールアドレスを設定してください。
```

Next.js の場合は、環境変数名を `NEXT_PUBLIC_ALLOWED_AUTH_EMAILS` に読み替えてください。

# redirect.html の白画面対策

Microsoftログイン実装で `public/redirect.html` を ssoSilent 用に使っている場合があります。

このファイルは基本的に軽量ページにしてください。ただし、ブラウザ本体が誤って `/redirect.html` に着地すると白画面になります。

Vite / React の SPA では、以下のようにトップレベル表示時だけアプリ本体へ戻すガードを入れると安全です。

```html
<!DOCTYPE html>
<html>
  <head>
    <meta charset="UTF-8" />
    <script>
      if (window.self === window.top) {
        window.location.replace("/");
      }
    </script>
  </head>
  <body></body>
</html>
```

hidden iframe 内の ssoSilent では `window.self !== window.top` なので、このリダイレクトは動きません。

# 白画面対策

認証まわりの実行時エラーで白画面にならないよう、可能であれば React の ErrorBoundary を追加してください。

最低限、以下のようなユーザー向けエラー画面を出してください。

```text
画面を表示できません
画面の表示中にエラーが発生しました。
```

# 検証手順

実装後、以下を確認してください。

## 1. build確認

```powershell
npm run build
```

PowerShell の実行ポリシーで `npm` が止まる場合は、Windows では `npm.cmd` を使ってください。

```powershell
npm.cmd run build
```

## 2. 未許可ユーザーの確認

許可メールを、実在しない値に一時変更します。

Vite の例:

```env
VITE_ALLOWED_AUTH_EMAILS=no-one@example.invalid
```

ローカルでは dev server を再起動してから確認してください。

```powershell
Ctrl + C
npm run dev
```

期待結果:

* Microsoftログイン自体は成功する
* アプリ本体は表示されない
* 拒否画面が表示される
* 判定対象メールと許可設定が表示される

## 3. 許可ユーザーの確認

拒否画面に表示された判定対象メールを、許可リストへ追加します。

例:

```env
VITE_ALLOWED_AUTH_EMAILS=user@company.onmicrosoft.com,user@company.com
```

ローカルでは再起動、本番では再デプロイして確認してください。

期待結果:

* Microsoftログイン後にアプリ本体が表示される

# 人間へ伝える内容

作業完了後、必要に応じて以下を人間へ伝えてください。

* Vite アプリでは `VITE_ALLOWED_AUTH_EMAILS` が必要です。
* Next.js アプリでは `NEXT_PUBLIC_ALLOWED_AUTH_EMAILS` が必要です。
* `.env` を変更しただけでは、起動中の Vite dev server には反映されません。
* 本番 Netlify では、環境変数変更後に再デプロイが必要です。
* Microsoft 側から `onmicrosoft.com` と独自ドメインの両方が返ることがあるため、拒否画面に表示された判定対象メールを見て、必要なメールを許可リストへ入れてください。

# ローカル開発時のトラブルシューティング

## ローカル開発環境で「fetch failed」エラーが出る場合

社内ネットワークやセキュリティソフト（プロキシ等）の影響で、API通信時に Node.js が自己署名証明書（SSL証明書）を「安全ではない」と判定し、バックエンドの通信が遮断されて `fetch failed`（自己署名証明書エラー）となる場合があります。

この問題は **ローカル開発環境のみで発生する特有のネットワーク制限** です。
回避するためには、ローカルの `.env` ファイルに以下を追記して、Node.js の証明書検証エラーを一時的に無効化してください。

```env
NODE_TLS_REJECT_UNAUTHORIZED=0
```

> [!WARNING]
> この設定はセキュリティ上の理由（中間者攻撃のリスク等）から、**本番環境（Netlify等）では絶対に設定しないでください。** 本番環境では正式なSSL証明書を用いた直接通信が行われるため、この設定は不要です。
